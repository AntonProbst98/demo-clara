# Collections Copilot

An internal tool for a B2B fintech collections team. Agents work a queue of
delinquent accounts one at a time — reading the account, applying the
**pre-computed** policy, generating a Spanish call script, and logging promises
to pay. Supervisors review the day's progress and portfolio health on a separate
dashboard.

Built with **Next.js 15 (App Router)**, **TypeScript**, **Tailwind CSS v4**, and
**Recharts**. Runs entirely on `localhost` — no deployment required.

---

## The one architectural idea that matters

> **Arithmetic and rules = code & data. Language = LLM.**

Every policy, discount, and dollar figure is **decided deterministically upstream**
(an n8n cleaning pipeline) and arrives in `data/cleaned_accounts.json`. This app
**consumes** those decisions — it never re-derives them. The Google Gemini LLM is
used for **exactly one thing**: turning an already-decided policy into
natural-language Spanish script text.

This is deliberate. An LLM must never be in the loop for a financial decision,
because it can hallucinate. It *is* well-suited to phrasing a script. The UI states
this boundary explicitly ("Deterministic · pre-computed" on the recommendation
zone), and the Gemini call is fenced behind a single API route with a
deterministic fallback so an agent is never blocked.

## Quick start

```bash
# 1. Install dependencies
npm install

# 2. (Optional) add your Gemini key so scripts are LLM-generated
cp .env.local.example .env.local
#    then edit .env.local and set GEMINI_API_KEY=...

# 3. Run
npm run dev
#    open http://localhost:3000
```

The app opens on the **Workspace**; the **Daily Progress** dashboard is one click
away in the header.

### Without a Gemini key

The app runs fully without a key. The `Generate script` button falls back to a
deterministic Spanish template and the UI labels the script's source
(`Generated by Gemini` vs `Template fallback`). No feature is blocked.

## Where the data goes

The app reads **`data/cleaned_accounts.json`** on the server at request time.

- A realistic **sample dataset is already included** so the app runs out of the box.
- To use your own data, **replace that file** with your pipeline's output — no code
  change and no rebuild needed (the pages are `force-dynamic`, so the file is
  re-read on every request).

Expected shape:

```jsonc
{
  "metadata": {
    "total_records": 57,
    "workable_records": 41,
    "needs_review_records": 13,
    "pending_review_records": 3,
    "duplicate_uuids_detected": 1,

  },
  "accounts": [
    {
```

## Configuring Gemini

Set the key in `.env.local` (read server-side only, never exposed to the browser):

```
GEMINI_API_KEY=your_key_here
```

- Model: `gemini-2.0-flash`, called via the REST API from
  `src/app/api/generate-script/route.ts`.
- The prompt is fed the **already-decided** amount, DPD, segment, and policy and
  is asked only to phrase a respectful, compliant Spanish script that guides toward
  a payment commitment (no threats, formal "usted").
- A 15s timeout and a template fallback guarantee the route always returns a usable
  script.

## Screens

### 1 · Account Workspace (`/workspace`)

- **Queue sidebar** — searchable and filterable by channel, segment, DPD bucket,
  and workable vs. needs-review, each filter showing a live count. Rows are
  color-coded by DPD and marked once contacted.
- **Zone 1 — Client account:** identity, prominent Amount Due, color-coded DPD
  (green < 30, amber 31–90, red > 90), utilization, and data-quality flags as chips.
- **Zone 2 — Negotiation recommendation:** the pre-computed policy, the rule that
  fired, and any upfront requirement. Needs-review accounts show
  *"Cannot recommend policy: data verification required"* with the blocking flags
  instead. Below, a `Generate script` button produces the Spanish script.
- **Zone 3 — Log promise to pay:** supports **multiple promises per account**, with
  an `Export all promises to CSV` action.
- **Zone 4 — Navigation:** Previous / Next through the filtered queue.

### 2 · Daily Progress (`/dashboard`)

- **KPI cards:** accounts contacted, promises logged, total committed value.
- **Promises by policy** (bar chart).
- **Data quality panel** from pipeline metadata, with a disposition bar.
- **Portfolio overview:** total exposure, exposure by channel, and DPD-bucket
  distribution — computed from the accounts array and colored by risk.

Logged promises live in a React context in the root layout, so they are shared
between the two screens across client-side navigation (no `localStorage` — this is
a demo).

## Design

The look targets a Brex / Ramp / Stripe-grade fintech aesthetic: ink on paper, one
reserved accent, a fixed status palette used only for meaning, tabular numerals for
money, and a strict spacing rhythm. The full brief — including the explicit
anti-references (no gradients, glassmorphism, gradient text, candy palettes,
side-stripe borders, or over-rounding) — is in [`PRODUCT.md`](./PRODUCT.md). The
chart palette is the validated, colorblind-safe set from the `dataviz` design
method. Both light and dark themes are supported.
